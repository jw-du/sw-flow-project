# SW-flow-project 项目规划

## 1. 项目目标

构建一个 Python 轻量 Agent Flow 系统，优先落地场景 1：

1. 用户提出需求后，系统优先检索已有 Flow。
2. 命中后展示候选 Flow 并请求用户确认。
3. 用户确认后执行 Flow，并反馈执行结果。

场景 2（未命中自动生成新 Flow）暂不进入首批开发。

## 2. 交付范围（MVP）

- Python 后端 API（FastAPI）
- Flow 检索与匹配模块
- Flow 执行引擎（顺序执行 + 基础依赖）
- 最小 Web UI（聊天输入、候选 Flow、执行结果）
- Docker 打包与一键启动

## 3. 里程碑

| 里程碑 | 输出 | 验收标准 |
|---|---|---|
| M1 核心骨架 | API、目录、数据模型 | 本地可启动健康检查接口 |
| M2 场景1检索链路 | Intent + Flow Matcher | 输入需求可返回候选 Flow |
| M3 场景1执行链路 | 执行器 + 结果回传 | 用户确认后可执行并返回结果 |
| M4 可试用交付 | Docker 化 + 文档 | 团队可通过 docker compose 启动 |

## 4. 工作分解

### 阶段 A：工程初始化

- 建立目录结构与 Python 包
- 配置依赖管理与环境变量
- 提供 Dockerfile 与 compose

### 阶段 B：场景 1 主链路

- 意图识别模块：提取任务意图与领域标签
- Flow 检索模块：规则检索 + 语义检索接口预留
- 候选 Flow 响应：返回 TopK 候选与匹配理由
- 用户确认执行：接收 flow_id 并执行

### 阶段 C：执行与可观测

- FlowSpec 解析
- Step 执行与上下文变量传递
- 执行记录落库（首版可用内存或 SQLite）

### 阶段 D：交付与演示

- 最小 Web UI
- Demo Flow 样例
- 部署与试用说明

## 5. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| Flow 匹配命中率不足 | 用户体验下降 | 先规则检索兜底，再叠加向量检索 |
| 执行器复杂度上升 | 迭代变慢 | 先支持顺序和基础依赖，map/并行后置 |
| 外部模型调用不稳定 | 执行失败 | 统一适配器 + 重试与超时策略 |
| Docker 环境不一致 | 演示失败 | 固化依赖版本并提供 .env.example |

## 6. 完成定义（DoD）

- `docker compose up` 可启动 API 和 UI
- 场景 1 能完整走通：提问 -> 候选 Flow -> 确认 -> 执行 -> 结果
- 项目文档包含：架构、目录、启动、扩展说明
