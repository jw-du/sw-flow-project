当前 `SW-flow-project` 的全链路串联逻辑和底层执行机制

### 一、 从前端到后端是怎么串联的？

整个系统的串联遵循标准的 **前后端分离架构**，基于 HTTP 协议进行通信。核心链路分为两个阶段：

#### 阶段 1：对话与推荐 (Match & Confirm)
1. **前端 (Web UI)**：
   - 用户在浏览器 (`ui/index.html`) 的聊天框输入需求（如“分析房地产”）。
   - 前端 JS 拦截输入，通过 `fetch` 发起一个 `POST /chat/query` 请求。
2. **后端 API (FastAPI)**：
   - `app/api/routes.py` 接收到请求。
   - 调用 `IntentRouter`：让大模型（DeepSeek）把“分析房地产”结构化，提取出关键词、领域标签以及参数（如 `{"analysis_topic": "房地产"}`）。
   - 调用 `FlowMatcher`：扫描 `flows/` 目录下所有的 `.md` 文件，根据提取的特征进行打分（关键词+1分、领域+0.8分等），选出得分最高的候选 Flow。
3. **前端渲染**：
   - 后端将候选 Flow 的元数据（包括 Mermaid 流程图源码和提取的参数）返回给前端。
   - 前端收到 JSON 后，动态生成一张“确认卡片”，并调用 `mermaid.js` 在卡片内渲染流程图，等待用户决断。

#### 阶段 2：确认与执行 (Execute)
1. **前端 (Web UI)**：
   - 用户点击卡片上的【确认并执行】按钮。
   - 前端把提取好的参数（`inputs`）和 `flow_id` 打包，通过 `fetch` 发起 `POST /flow/execute` 请求。
2. **后端 API (FastAPI)**：
   - `routes.py` 接收请求，首先在 `execute_output/` 下创建一个**时间戳专属文件夹**，并将 `output_path` 指向该文件夹以实现资产隔离。
   - 然后，把 `flow_id` 和修改后的 `inputs` 丢给核心引擎 `FlowExecutor`。
   - 引擎跑完后，将各个步骤的状态和最终生成的 Markdown 文本返回给前端。
3. **前端渲染**：
   - 前端拿到结果，使用 `marked.js` 将 Markdown 渲染为漂亮的富文本，展示在聊天气泡中。

### 二、 选中 Flow 后，执行机制是怎么样的？

当你点击执行后，接力棒交给了 **Flow 执行引擎** (`app/flow_engine/executor.py`)。它的运行机制类似于一个微型的状态机或 DAG（有向无环图）调度器，核心机制如下：

#### 1. 解析执行计划
引擎通过 `FlowRepository` 去读你选中的 `.md` 文件（如 `market-environment-china.md`），并提取出里面的 ````json` 代码块。这个 JSON 数组就是**执行步骤 (Steps)** 的蓝图。

#### 2. 建立全局上下文 (Context)
引擎在内存里开辟一个大字典 `context`：
```python
context = {
    "inputs": {"analysis_topic": "房地产", "lookback_days": 7},
    "steps": {} # 用来存放每一步的输出
}
```

#### 3. 顺序步进与变量渲染 (Template Resolver)
引擎开始按 JSON 数组的顺序遍历每一个 `step`。
在执行某一步（比如 `bing_search`）之前，引擎会先做**变量替换**：
- 它看到参数里写着 `"{{inputs.analysis_topic}}"`，就会去 `context` 里找，把它替换成真实的字符串 `"房地产"`。
- 如果看到 `"{{keyword_step.queries}}"`，就会去拿上一步的输出结果。

#### 4. 适配器分发 (Adapter Registry)
变量替换完成后，引擎看到了 `skill: "bing_search"`。它不知道这是什么，但它会按优先级去找**适配器 (Adapter)**：
- **查 Python 脚本**：它先去 `skills/bing_search/scripts/` 目录下找有没有 `search.py`。如果有，就触发 `PythonScriptAdapter`，通过命令行启动该脚本，把参数传进去，并捕获 `stdout` 的 JSON 结果。
- **查 LLM Prompt**：如果没找到脚本，它再去查 `skills/industry-keywords/SKILL.md`。如果存在，就触发 `LlmPromptAdapter`，把该文件的内容作为 System Prompt，把你的参数作为 User Message 发给大模型，并尝试将模型的回答解析为 JSON。
- **查内置函数**：如果是 `builtin.echo` 或 `builtin.json`，就直接走 Python 原生逻辑处理。

#### 5. 状态记录与 Map 循环
- **普通步骤**：执行完后，结果会被塞回全局 `context` 里，供下一步使用。
- **Map 步骤**：如果在 JSON 里声明了 `type: "map"` 且提供了 `items: "{{keyword_step.queries}}"`，引擎会**开启一个循环**，遍历 `items` 里的每一条数据，分别执行一遍对应的 Skill，最后把所有结果打包成一个大数组塞回 `context`。

直到所有步骤走完，或者中间某一步报错（立即中断并返回错误），引擎将最终的 `context` 返回给 API 层。
